# Minesweeper Online Assistant

> 本脚本只做高亮提示，**不发送任何点击/键盘事件**，不会操作网页棋盘状态。

这是给 <https://minesweeper.online/> 用的扫雷辅助用户脚本。

常规分析会读取网页棋盘 DOM，不做截图识别；死猜救援会在目标格本身没有确定结论时读取当前对局已捕获的答案源，并且只提示用户指定的一格。救援必须在当前局有可验证答案源时才可用；答案源可以来自开局/加密下发的完整雷图，也可以来自用户踩雷后网站暴露的失败雷图。当前实现支持：

- 绿色 `OK`：确定安全，可以打开。
- 红色 `M`：逻辑上 100% 确定是雷，可以插旗。
- 绿色描边旗子：用户已插旗，并且算法也确认该格是雷。
- 黄色半透明 `?`：用户已插旗，但算法尚未确认该格是雷，方便复查。
- 蓝色百分比：开启"概率"后显示雷率；全局雷数可靠时使用全局加权，否则小组件优先使用精确枚举，超限时退回估计，只作参考。
- 救援 `开` / `避`：目标格本身没有确定 `OK` / `M` 结论、且当前局有可验证答案源时可用；如果用户本局已经踩雷并通过网站继续玩，脚本会缓存失败时暴露的雷图；每局最多检查 3 个不同的临时旗标记格。

## 安装

1. 在浏览器安装 Tampermonkey 或 Violentmonkey。
2. 打开安装地址：
   <https://raw.githubusercontent.com/SeptYagu/minesweeper-online-assistant/main/minesweeper-online-helper.user.js>
3. 脚本管理器弹出安装页后确认安装。
4. 打开 <https://minesweeper.online/>，确认右下角出现"扫雷辅助"面板后，再开始新的一局游戏。

脚本头已配置 `@updateURL` / `@downloadURL`，后续只要 GitHub 上的 `@version` 变大，Tampermonkey 会按自己的更新周期检查并更新。

## 使用

- "分析"：重新读取棋盘并高亮推理结果。
- "清除"：移除辅助器高亮。
- "救援"：目标格本身没有确定结论时，先把一个死猜格临时插旗，再点击；每局最多检查 3 个不同格，只会标记当前目标格。按钮变绿表示当前局已有可用答案源且次数未用完；按钮变红表示没有可用答案源或次数已用完。若显示 `开`，先取消这个临时旗再打开；若显示 `避`，保留旗。普通局开局通常没有完整雷图；如果本局已经踩雷并使用网站的继续玩，脚本会使用失败时缓存的雷图。
- "自动"：棋盘变化后自动重新分析。
- "概率"：显示未确定格的雷率，只显示，不做任何操作。
- "说明"：控制推理说明模块和悬停解释高亮。
- 把鼠标移到 `OK` / `M` 上：面板显示该结论的推理来源，并用额外描边标出相关数字格和候选格；说明会保留到下一次真正分析或清除。
- 按 `~` / 反引号键：显示或隐藏当前分析高亮；输入框里不会触发。

面板底部会显示一行提示，说明基础标记、救援使用前提、答案源限制，以及本脚本不会发送任何鼠标事件。

## 行为边界

**会做**：

- 读取当前对局棋盘的 DOM 状态（开 / 旗 / 数字）。
- 可靠时读取顶部剩余雷数，并把它换算为全局总雷数约束。
- 在网页上叠加 CSS 高亮（绿/红/黄/蓝）与对应标签。
- 悬停在确定结论上时，显示基础数字约束、集合差、精确枚举或全局雷数推理说明。
- 对目标格本身没有确定结论的临时旗标记使用死猜救援；每局最多 3 个不同格，且只显示这一格是可开还是应避开。
- 死猜救援只被动使用本页捕获到的当前局答案源；如果网站没有下发完整雷图、且本局尚未踩雷暴露雷图，脚本不会猜测或补全。
- 如果用户踩雷后网站允许继续玩，脚本会保存这次失败响应里暴露的当前局雷图；之后同一局可用它做最多 3 个目标格救援。
- 监听棋盘 DOM 变化，触发重新分析。
- 在 `localStorage` 保存用户的"自动"、"概率"、"说明"三个开关、折叠状态，以及每局救援计数。

**不会做**：

- 不发送任何鼠标 / 键盘事件。
- 不修改网页内部的棋盘状态（不开格、不插旗、不改数字）。
- 不调用任何外部网络资源。
- 不使用 CDP / WebDriver / 原生驱动。
- 不读取网页之外的用户状态。
- 不批量展示整局答案，不透露非目标格答案。
- 不补读取脚本安装/更新前已经错过的历史 websocket 初始化包。
- 不使用截图、OCR 或图像识别。

## 救援可用条件

点击"救援"时，脚本会重新读取棋盘，并依次检查：

- 脚本已捕获到当前局初始化数据，或已捕获到当前局踩雷失败响应。
- 当前对局有可验证的答案源：完整初始化雷图、可解码 `lpe` 雷图，或失败后网站暴露的完整雷图。
- 用户已经临时插旗标记了一个目标格；如果有多个未确认旗，会优先使用最近刚插的旗，否则要求只保留一个救援标记。
- 这个目标格本身没有确定 `OK` / `M` 结论。
- 本局救援次数还没超过 3 个不同格。

如果答案源不可用，按钮会尽量显示具体原因：例如未捕获到当前局、网站未下发完整雷图、加密雷图未解码、答案包格式不支持，或首开触发过雷位重排。更新脚本后已打开的旧页面不会补捕获过去的初始化包；刷新正在进行中的对局时，脚本只能捕获刷新后服务器重新下发的数据，如果这份数据仍不是完整雷图、且本局没有失败缓存，会继续拒绝救援。

救援按钮颜色只表达当前局答案源和次数状态：绿色表示已拿到可用雷图且还有救援次数，红色表示没有可用雷图或救援次数已用完。即使尚未临时插旗目标格，已拿到雷图时按钮也会保持绿色，方便先确认答案源状态。

踩雷失败后，minesweeper.online 会在失败响应里下发全部雷位用于渲染失败棋盘。脚本只在这件事已经发生时缓存当前 `gameId` 的雷图；如果用户随后使用网站的继续玩/回退一步功能，救援可以使用这份缓存回答用户临时标记的单个目标格。脚本不会主动制造失败，也不会展示整局答案。

2026-06-22 真实浏览器验证结果：标准 Beginner 新局的 websocket 初始化响应只下发全 0 的 `t/o/f/g/g2` 数组，首开响应只返回已打开区域的 `touchCells`，没有被动暴露完整雷图；页面上的 `window.h` 也不是可用的全局运行时棋盘对象。因此在这类对局中看到"网站未下发完整雷图（已识别 0/10 雷）"是当前能力边界，不是用户操作错误。

救援现在按"目标格是否死猜"判断，不再要求全盘没有其它确定结论。

## 技术说明

脚本适配 minesweeper.online 当前的棋盘结构：

- 格子 ID 是 `cell_x_y`。
- 格子带有 `data-x` / `data-y`。
- 状态通过类名表达，例如 `hd_closed`、`hd_flag`、`hd_type1` 到 `hd_type8`。

为了兼容换皮肤，脚本按类名后缀识别状态，不固定依赖 `hd_` 这一种前缀。

求解器在 `solveBoard` 里做约束传播：每个已开数字格构成一条"邻居里还有几颗雷"的约束；用基础规则（约束数为 0 ⇒ 全安全，约束数等于集合大小 ⇒ 全雷）+ 集合差规则（A ⊂ B ⇒ B\A 的雷数 = countB − countA）迭代直到不动点。求解本身是确定性的，不会把用户已插的旗当作"真实有雷"前提，避免手动插错旗后污染后续判断。

高亮通过注入一个 `<style id="msah-style-${salt}">` 标签实现，类名形如 `msah-${salt}-safe`。面板 ID 是 `msah-panel-${salt}`，避免与站点其它代码冲突。
